# 铁壁 v0.5.0 收尾方案 — 对抗审查报告

---

## 攻击点 #1: Recall 提升是打地鼠，不是系统性修复

**为什么是问题：**

26个FN，方案只挑了4个模式来修。选这4个的依据是什么？是占了26个FN的80%还是20%？方案里没有优先级分析，看起来就是"哪些好修修哪些"。

"函数调用间传播"尤其危险——这是SAST领域的老大难问题。用纯AST做跨函数taint追踪，你需要至少解决：同一文件内的简单调用、跨文件导入、类方法调用（多态怎么办？）、递归函数、高阶函数（`map(filter(tainted))`）。方案里对这个完全没有边界定义。如果边界不清，要么两周做不完，要么做出来的是一个到处漏的假方案。

另外，列表索引 `values[0]` 的追踪——如果代码是：
```python
values = request.form.getlist()
for v in values:
    sql.execute(v)  # 这个循环里的v你能追踪吗？
```
你只追踪 `values[0]` 的字面量下标，那 `for v in values` 呢？`values[i]` 用变量做下标呢？这些比字面量下标更常见。

**建议：**

1. 先对26个FN做一次分类统计，算出每种模式占多少比例
2. 按覆盖率排序，优先修占比最高的2-3个
3. "函数调用间传播"拆成子任务：先做同文件内单层调用（限制scope），标注清楚什么不做
4. 不要定"Recall 0.55+"，而是定"FN减少X个"——Recall分母会随新项目变化

## 攻击点 #2: 真实项目验证选型有致命缺陷

**为什么是问题：**

**CTFd 完全不适合测 FP 率。** CTFd 是一个 CTF（Capture The Flag）竞赛平台，它*故意*包含安全漏洞作为比赛题目。你把SAST工具跑在上面，扫描出来的告警里，哪些是真漏洞（比赛题目）、哪些是框架代码的安全写法、哪些是真正的误报——你分得清吗？除非你逐行审计过CTFd的代码，否则没有ground truth。

没有ground truth就没有FP率。你得到的只是一个数字，不知道它是什么意思。

fiber 128K行Go也是类似问题。方案里没提Go管线的当前状态。如果Go的AST模式和规则集还不如Python成熟，那跑128K行基本上就是噪音生成器。

**建议：**

1. Python验证换成已知标注数据集：OWASP Benchmark、CWE-Compatible的测试套件，或者至少选一个你熟悉代码、能手工标注的中型项目
2. Go管线先在小范围（比如fiber的examples目录）验证规则准确性，再上全量
3. 如果非要用真实项目，必须先建立ground truth的标注计划（至少抽100个告警手工审计）

## 攻击点 #3: AI 引擎 79% 失败率不是 retry 能解决的

**为什么是问题：**

19个batch只完成4个，失败率79%。加retry/backoff解决的是偶发超时和临时限流，解决不了系统性失败。

79%的失败率意味着大概率是以下之一：
- API的rate limit太严格（retry只会加剧，因为重试也消耗quota）
- batch size太大导致超时（需要减小batch，不是重试）
- 请求格式本身有问题导致API拒绝
- DeepSeek API本身不稳定

不诊断根因就直接上retry，结果可能是：原来79%失败+重试后仍然失败+消耗更多token+更慢。

而且方案里说要测"DeepSeek对真实项目的FP抑制率"——FP抑制的前提是AI能区分TP和FP。如果DeepSeek准确率不高：
- 把TP判成FP → Recall降低（跟方向1冲突）
- 把FP判成TP → FP率不变

没有baseline数据支撑"AI抑制FP"这个假设。

**建议：**

1. 先查那15个失败batch的具体错误码（超时？429？5xx？）
2. 如果是限流，减小batch size + 加间隔，不是加retry
3. 先在小数据集上测DeepSeek的TP/FP判别准确率，建立baseline，再决定是否启用
4. 加 `analysis_status` 字段是对的，但应该在修复根因之后

## 攻击点 #4: 1.5-2天的时间估算严重低估

**为什么是问题：**

四个方向里至少有两个有显著不确定性：
- **函数调用间传播**：历史上无数SAST工具在这个问题上花了人月甚至人年。即使用最简单的方案（同文件单层调用），从设计到实现到调试，1天是最低估计。
- **真实项目验证**：跑一次128K行Go + CTFd + 手工分析结果，半天就没了。如果结果不理想（大概率），还要调规则、再跑、再分析。
- **AI引擎**：如果根因不是retry能解决的，可能要重构批处理逻辑。

加起来在乐观情况下也要2天。而且Brain A的方案里没有写如果某个方向卡住了怎么办——是砍掉还是延期？

**建议：**

1. 给四个方向排优先级：Recall修复 > 提交发布 > 真实验证 > AI引擎
2. 时间到了砍低优先级方向，不要全都要
3. 函数调用间传播设硬性时间盒（比如4小时），到时间不管做到什么程度都收

## 攻击点 #5: 缺少回归测试计划

**为什么是问题：**

修AST模式可能引入新的FP。比如加了"列表索引源"的taint追踪后，所有 `some_list[0]` 都可能被标记为tainted，包括 `config_list[0]` 这种安全的用法 → FP激增。

Brain A的方案里完全没有提到回归测试。这意味着改了AST分析逻辑后，没办法自动验证"之前能检测的漏洞现在还能检测，之前不会误报的现在不会误报"。

**建议：**

在 `ironwall_custom_scanner.py` 的修改同时，维护一组最小回归用例（fixture文件），包含已知TP和已知安全代码，每次修改后跑一遍。

---

## 总结判断

| 维度 | 评价 |
|------|------|
| Recall修复 | 方向对，但选点缺乏数据支撑，函数调用间传播scope不明确 |
| 真实验证 | 选型有问题，CTFd完全不适合，缺少ground truth计划 |
| AI引擎 | 根因不明就上retry是浪费token，应先诊断 |
| 时间估算 | 乐观了至少2倍 |
| 回归测试 | 缺失 |

**总体：方案方向没问题，但执行细节不扎实。最大风险是"函数调用间传播"失控拖垮整个时间表，以及AI引擎在根因不明的情况下重试烧token。建议把四线缩减为两线——修Recall + 提交发布——另外两个挪到v0.6.0。**
