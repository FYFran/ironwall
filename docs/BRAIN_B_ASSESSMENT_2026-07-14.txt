## 一、现在什么水平？

**一句话：Demo++ 级，离产品级还差一个"外部验证"。**

| 维度 | 得分 | 说明 |
|:-----|:---:|:-----|
| **Precision（精确率）** | ⭐⭐⭐⭐ | 94.4% on vulnbench（17TP/1FP）。但 vulnbench 的测试集是自己构建的——这是循环验证，不能对外声称 |
| **Recall（召回率）** | ⭐⭐ | 81% on go-vuln-app，但同样是自造靶场。底层 SAST 引擎的 recall 上限只有 7-27%（Bennett 2024），Phase B 的 MISSING 控制检测是 unique value，但覆盖面太小 |
| **差异化** | ⭐⭐⭐⭐⭐ | 4-step AI 解释攻击路径 + 修复方案 + MISSING 控制检测。业界没有直接竞品做"AI 审计 AI"这件事 |
| **工程完备性** | ⭐⭐⭐ | 编译全绿，offline 可用，cost $0.016/scan。但 call graph 存在却未接入 TRACE——等于发动机造好了没装轮子 |
| **可信度** | ⭐ | **零外部验证。** 没跑过 OWASP Benchmark、没跑过 NIST SAMATE、没在真实 CVE 上验证、没第三方审计 |

## 二、业界怎么评定 SAST 好坏？

业界的评定体系分三层：

### 第一层：标准 Benchmarks（学术/政府采购必备）

| Benchmark | 覆盖语言 | 地位 |
|:----------|:---------|:-----|
| **OWASP Benchmark** | Java（主力） | 事实上的行业标准。每个 SAST 厂商必跑 |
| **NIST SAMATE** | C/C++, Java | 美国政府采购参考标准 |
| **Juliet Test Suite** | C/C++, Java, C# | NIST 维护，61,000+ 测试用例 |
| **CVEfixes / Big-Vul** | 多语言 | 真实 CVE 数据集，现代学术界首选 |

**铁壁现状：一个都没跑。**

### 第二层：2026 年新 Benchmark（LLM + SAST 交叉验证）

| Benchmark | 核心发现 |
|:----------|:---------|
| **RealSec-bench (2026)** | Hybrid SAST+LLM: F1 0.91-0.99, FPR 10-17% |
| **RealVuln (2026.4)** | 15 个扫描器测 26 个 Python 仓库，最好 F3=73（Kolega.Dev），Semgrep=18，LLM=52 |
| **VulnBench JS (2026.6)** | 最好 LLM F1=75%（Opus 4.6），非确定性：50% 的 findings 只在 1/5 运行中出现 |

### 第三层：实战指标

- **时间到利用**: CVE 2025 平均 <5 天即被利用
- **检出率现实**: 单个 SAST 工具 11-27%，4 个组合 39%，+自定义规则 45%（Bennett 2024）
- **误报率**: 行业级 SAST 通常 30-60% FPR。铁壁 94.4% precision 如果能在外部测试集复现，属于顶级

### 核心指标定义

```
Precision = TP / (TP + FP)    ← 报出来的东西里有多少是真的
Recall    = TP / (TP + FN)    ← 真实漏洞里有多少被发现了
F1        = 2*P*R / (P+R)     ← 综合评分
F3        = 优先召回率
```

**铁壁当前的致命伤：以上所有指标都只在自造靶场上测过。**

## 三、真的能查出问题吗？

**能查出一部分，但查不全。**

### 能查出的（已验证）
- ✅ **CSRF 缺失控制** — campus_go 18 条 HIGH 全中（Phase B MISSING 的强项）
- ✅ **明显的 SAST 规则覆盖** — semgrep+gosec+bandit 三层叠加，硬编码密钥、SQL 注入、不安全依赖等
- ✅ **CWE-306 缺失认证** — GT-008 经过 Phase B VERIFY 验证确认

### 查不出的
- ❌ **需要跨文件数据流分析的业务逻辑漏洞** — call graph 造好了但 TRACE 没接入
- ❌ **非 Gin/Flask 框架的漏洞** — 框架覆盖只有 4 个
- ❌ **运行时漏洞** — 没有动态分析（Neo 做这个，$ enterprise）
- ❌ **跨服务/供应链深层漏洞** — Arnica 做这个

### 可信度问题（严重）

> **go-vuln-app 的 16 个漏洞是自己定义的 → 81% 检出率是循环验证。**
>
> 类比：自己出题、自己答卷、自己批改 → 81 分不能对外说。

学术界的正确做法是：在 **CVEfixes** 或 **Big-Vul** 数据集上跑，这些是真实 CVE 提取的代码片段，和你的训练数据（如果有的话）没有重叠。
