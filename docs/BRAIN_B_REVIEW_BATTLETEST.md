# 铁壁大考方案 — 红队攻击

---

## 攻击点 1：数据泄露 — OWASP Benchmark 已死

> **Phase 2 用 OWASP Benchmark 做"独立"对比，是方案最大的结构性问题。**

OWASP Benchmark 是**公开、静态、众所周知**的测试集。任何 SAST 厂商只要做过一次 benchmark 驱动的调优——哪怕无意识的——就已完成数据泄露。这不是"铁壁有没有主动作弊"的问题，而是**benchmark science 101**：公开测试集一旦被用于开发反馈循环，F1 就不再反映真实能力。

更糟的是，OWASP Benchmark 本身已被学术界多次批评[¹](#ref-1)：
- 测试用例是**合成代码**，不是真实项目中的漏洞模式
- 漏洞/非漏洞比例人为平衡（50/50），与真实代码分布（通常 <1% 含漏洞）完全脱节
- 真实代码中的"非漏洞"比 benchmark 的非漏洞复杂得多——假阳性率在实战中会**急剧膨胀**

**结论**：Phase 2 的 OWASP Benchmark 对比不能被称为"独立benchmark"。如果铁壁曾用 OWASP Benchmark 做过任何开发决策，这个数字对闲鱼买家是**误导**。

---

## 攻击点 2：假阴性不可测 — 整个方案只知"找到什么"，不知"漏了什么"

> **这是四个 Phase 共同的致命盲区。**

Phase 1 测 CVE：验证铁壁能否检测到**已知**漏洞。好，能检测到。然后呢？没检测到的怎么办？**你不知道铁壁漏了多少。** CVE 基准只能测召回率在已知漏洞上的表现，不能测在未知漏洞上的表现。

Phase 3 更糟。模拟"客户流程"——扫描陌生项目，手工审计 TOP 20 告警。这只能告诉你：
- 前 20 个告警里有多少是真阳性 ✓
- **完全不知道**有多少真漏洞没有出现在 TOP 20 里 ✗
- **完全不知道**有多少真漏洞铁壁根本没检测到 ✗

要测假阴性，你需要：一个人**全量手工审计**项目建立 ground truth，然后看铁壁漏了多少。而方案明确说"审计 TOP 20 告警"——这不是审计，这是**在铁壁的输出里找真阳性**。

**打个比方**：这就像测试机场安检——只检查安检拦下的人里有多少真带违禁品的，但从不派人带违禁品闯关测漏检率。然后声称"我们的安检准确率很高"。

---

## 攻击点 3：审计独立性崩塌 — "双人交叉验证"是伪独立

> **Brain A 标记 + Codex 验证 ≠ 独立审计。**

问题链：
1. Brain A 是铁壁的开发者，对铁壁的弱点和盲区有**根深蒂固的先验信念**
2. Brain A 选择 TOP 20 告警来做审计——**选择本身就有偏差**（高分告警天然偏向铁壁擅长的模式）
3. Codex 验证时，Brain A 提供了上下文、告警描述、代码片段——这不是盲审，Codex 的注意力已经被 Brain A 引导到特定位置和特定判断上
4. 两人都不是**安全中立的第三方**。Brain A 有利益相关（闲鱼数据），Codex 被 Brain A 的提示词框架约束

真正的独立审计需要：
- 审计者不知道工具输出
- 审计者在全量代码上找漏洞
- 审计者与工具开发者无利益关系
- 交叉验证使用盲法（blinding）

---

## 攻击点 4：样本量 = 噪声 — 3 个 CVE + 1 个项目什么也证明不了

> **统计上，这个方案产出的数字没有意义。**

- **3 个 CVE**：即使 3/3 检出（Recall=100%），95% 置信区间是 [29%, 100%]。**下限 29%。**
- **1 个陌生项目**：不管检出率是多少，样本量 n=1，不能泛化。一周后换一个项目，数字可能完全不同。
- **Phase 2 对比维度爆炸**：3 工具 × 2 模式（有/无 AI）× 多个指标，每个格子里样本量被摊薄到几近于零。

**你能想象一个安全产品在闲鱼上说"经过实战检验，我们在 3 个 CVE 上检出率 100%"吗？** 买家随便搜一下 CVE Details 就知道你在说什么。

---

## 攻击点 5（Bonus）：发表偏差 — 商业动机污染实验设计

> **方案目标明确写了"产出可用于闲鱼listing的实战数据"。**

这意味着实验从一开始就不是为求真，而是为**产生好看的数字**。每一个方法论缺陷都会被这个动机放大：

- CVE 选哪 3 个？——潜意识会选铁壁擅长的
- 陌生项目选哪个？——会选代码风格接近铁壁训练数据的
- 数据不好看怎么办？——方案里没有预设"数据不合格就不发闲鱼"的闸门
- "诚实标注强弱项"由谁来审核？——没有独立审核机制

科学研究中的**发表偏差（publication bias）**在这里完全适用：只有好看的数字才会出现在闲鱼上。

## 总结判断

| 维度 | 评分 | 致命问题 |
|------|------|----------|
| 内部一致性 | ⭐⭐⭐ | 流程逻辑自洽 |
| 统计有效性 | ⭐ | 样本量近乎噪声 |
| Ground truth | ⭐ | 假阴性完全不可测 |
| 独立性 | ⭐ | 开发者自测，无盲审 |
| 可泛化性 | ⭐ | 公开 benchmark + 1 个项目 |
| 诚实性风险 | ⭐ | 商业目的驱动，无独立闸门 |

**总评：这个方案测不出铁壁在真实代码上的真实表现。它测的是"铁壁在自己擅长的那几个点上表现如何"。**

如果真想拿出经得起挑战的实战数据，至少需要：
1. **封闭测试集**：从未用于铁壁开发的第三方漏洞数据集（如 NVD 中筛选铁壁从未接触过的 CVE 时间窗口）
2. **假阴性测量**：对至少一个项目做全量手工审计建立 ground truth
3. **盲法审计**：第三方安全研究员（不知道工具身份）交叉验证
4. **预设发表标准**：实验开始前就声明"无论结果如何都发布"，否则选择性报告无解

否则，这份"铁壁大考"报告发到闲鱼上，随便一个懂安全的买家问一句"你们假阴性率多少"——方案就塌了。
