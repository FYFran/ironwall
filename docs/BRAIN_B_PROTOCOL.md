# Brain B 审查协议 v1.0

> 所有设计决策必须通过领域顶级专家的对抗审查。
> 不通过Brain B审查的设计，不允许写代码。
> 本协议适用于所有项目，永不过期。

---

## 什么是Brain B审查

Brain B是一个**虚拟角色**——该领域的顶级专家，具有：
- 顶级视野（了解全球最新研究和产业动态）
- 顶级眼界（见过所有失败模式）
- 顶级能力（第一性原理思考，不被表象迷惑）

Brain B的**唯一职责**：攻击Brain A的设计，直到要么设计被推翻修正，要么Brain A用更强论据说服Brain B。

**规则**：
1. Brain B至少提出3个攻击点
2. Brain A必须逐一回应
3. 有分歧时，Brain B的意见优先（保守原则）
4. 最终必须达成共识，不允许"同意分歧"
5. 共识结果写入设计文档，标注"双脑共识通过"

---

## 审查流程

```
1. Brain A 提出设计方案
      ↓
2. Brain B 审阅 → 提出攻击点（至少3个）
      ↓
3. Brain A 逐一回应
      ↓
4. 如有分歧 → 继续辩论，直到收敛
      ↓
5. 双脑共识 → 修正设计 → 写入文档
      ↓
6. 设计文档头部标注 "双脑共识通过 | YYYY-MM-DD | Brain B: [领域]专家"
```

## 各领域Brain B配置

不同问题领域，调用不同专家配置：

### 安全架构设计
- 角色：应用安全架构师，15年经验
- 专业知识：零信任架构、威胁建模、OWASP、CWE/CVE体系
- 审查重点：攻击面是否完整？边界条件？绕过路径？

### ML/AI系统设计
- 角色：ML系统工程专家 + 安全研究员
- 专业知识：LLM评估、prompt工程、模型幻觉、对抗攻击
- 审查重点：prompt是否可测试？是否有量化评估？降级策略？

### 产品/商业决策
- 角色：产品VP，做过0→1→100
- 专业知识：PMF验证、定价策略、GTM、用户研究
- 审查重点：客户真的会付钱吗？为什么是你不是别人？

### 工程架构设计
- 角色：Staff Engineer，维护过百万行代码库
- 专业知识：架构模式、技术债务、可维护性、错误处理
- 审查重点：复杂度是否合理？未来会后悔什么？单点故障？

### 性能/基础设施
- 角色：SRE/性能工程师
- 专业知识：分布式系统、容量规划、故障模式
- 审查重点：瓶颈在哪？爆炸半径？监控覆盖？

## 铁律

1. **任何非trivial设计必须过Brain B审查。** Trivial定义：单文件、单函数、不影响架构、不改变接口。
2. **Brain B不能说"好，可以"。** 必须找到至少1个问题。如果真找不到 → 换个角度再找。
3. **Brain A不能敷衍回应。** "你可能是对的"不是回应。必须说"接受/驳回/妥协" + 具体理由。
4. **审查记录必须存档。** 存在设计文档同级目录，文件名: `REVIEW_[设计名].md`。
5. **新会话开始时，如果发现上次设计没有Brain B审查，先补审查，再写代码。**

## 为什么这个协议必须存在

我们在Ironwall v0.5.0设计过程中犯过这些错误：
1. 设计了6个模块，3个被Brain B砍掉（过度设计）
2. ContextProvider固定30行→Brain B指出应该整个函数体（预设窗口会漏跨行漏洞）
3. Attacker用LLM验证LLM→Brain B指出自洽偏差问题
4. 沉浸架构忘记赚钱→Brain B纠正优先级

**如果没有Brain B，这些错误会进入代码，浪费3-5天后才发现。**

Brain B不是锦上添花，是工程质量的最后防线。
