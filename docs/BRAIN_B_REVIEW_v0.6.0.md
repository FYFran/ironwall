# 🔴 Brain B 攻击分析：v0.6.0 方案

---

## 攻击 1：79% 失败率当工程问题修，不做根因诊断

**问题：**

Brain A 列了一堆"可能修复"——batch size 25→10、加间隔、超时 300→600s——全是猜测。79% 失败率有三种完全不同的根因可能：

| 根因类型 | 表现 | Brain A 的修复是否有效 |
|---------|------|---------------------|
| DeepSeek API 限流/并发限制 | 429/503 | 部分有效（batch size + 间隔） |
| Prompt 过长导致 token 截断/拒绝 | 400/context overflow | **完全无效** |
| 模型对特定代码模式输出格式错误，解析器崩溃 | 非HTTP错误，解析失败 | **完全无效** |

0.5 天不给诊断只给猜测，修错了就是浪费 0.5 天。SAST 产品的 AI 引擎失败，第一件事是**分桶统计失败原因**——按错误码、按文件大小、按语言、按 prompt token 数。你现在连失败是 400 还是 429 还是 timeout 都说不清楚。

**建议：**

先用 2 小时做失败分桶——拉 benchmark_50 的全部失败日志，按 HTTP 状态码 + 异常类型分类。只有看到数据才决定修什么。别猜。

## 攻击 2：成功标准是"能跑完"，不是"跑得对"

**问题：**

提案原话：**"验证标准：能完整跑完 benchmark_50（50文件）。不追FP抑制率数字，先追能跑完。"**

这是我见过的 SAST 产品最危险的验证策略。一个 79% 失败率的系统，"修到能跑完"最可能的结果是什么？——把原来 crash 的 case 全部变成**静默错误输出**。返回空结果、返回格式错误的 JSON、返回幻觉结论。系统不 crash 了，但输出不可信。

更致命的是：闲鱼 listing 上写的是 **"AI分析帮你判断哪些漏洞真该修准确率94.4%"**。你的 F1 现在是 0.568。v0.6.0 的验证完全不追准确率数字，那你凭什么把这个产品卖给用户？

**建议：**

成功标准必须是 **F1 ≥ 某个基线**，不能只是"能跑完"。至少拿 benchmark 做一次 before/after 对比。如果你不敢看 after 的 F1 数字，说明你对修复方案没信心。

---

## 攻击 3：三个"真实项目"选品是精心设计的营销材料

**问题：**

三个项目逐个看：

| 项目 | 性质 | 为什么选它 | 在闲鱼 listing 里会被怎么用 |
|------|------|-----------|--------------------------|
| fiber (Go, 128K行) | 已知结果的复现测试 | "复现旧数据23→1" — 已有预期 | 不会出现在 listing 里（数字不好看） |
| vulnpy (Python) | **故意埋漏洞的靶场** | 检测率必然高 | "检出率 XX%，远超竞品" |
| Flask-Security (Python) | **成熟安全库** | FP 率必然低 | "误报率仅 X%，业界最低" |

这是 SAST 厂商做 benchmark 的经典套路：用靶场测检测率，用干净项目测 FP 率，两个数字分开报。用户以为说的是同一个项目，实际上不是。

vulnpy 是故意埋的漏洞，和真实业务代码的漏洞分布完全不同——真实代码的漏洞是副作用，靶场的漏洞是刻意构造的。在靶场上测出的检测率没有外推价值。

**建议：**

选品规则改成：**两个真实业务项目（和闲鱼客户的实际代码栈接近）+ 一个 benchmark 数据集**。靶场项目的结果单独标注"靶场数据，不代表真实检测率"。Flask-Security 这种安全库也标注"项目本身经过安全审计，FP 率不代表一般项目"。

---

## 攻击 4：94.4% 准确率声明与 F1=0.568 之间存在数据断层，v0.6.0 填不上

**问题：**

这是最严重的。目前 F1=0.568（benchmark 数据）。闲鱼 listing 写的 94.4% 准确率。这两者之间的差距你没解释过，v0.6.0 也没有任何计划去弥合。

F1=0.568 意味着在 benchmark 上——precision 和 recall 的综合值只有 56.8%。哪怕你把"准确率"重新定义为别的指标（比如"AI 判断是否需要修的准确率"，不包含检测环节），你也没有提供这个指标的计算方法和数据来源。

如果有人买了 ¥99-599 的服务，跑完发现 AI 分析不准，要求退款——你拿什么数据证明 94.4%？

**建议：**

在 v0.6.0 里明确：94.4% 是从哪个数据集、用哪个指标、在哪个版本上算出来的。如果算不出来，**立即从闲鱼 listing 撤下这个数字**。虚假宣称在安全产品上是信任自杀。

---

## 总判断

v0.6.0 不是一个安全产品的迭代计划，是一个**从"不可用"到"看起来可用"的 rush job**。三个核心问题：

1. **不诊断就修** → 可能修错方向，浪费唯一的 0.5 天 AI 引擎预算
2. **不追正确性** → "能跑完"不等于"跑得对"，修完可能产出更隐蔽的错误
3. **选品服务于营销** → 三个"真实项目"的结论没有外推价值，但会被用来支撑 listing 上的数字

**一句话：这方案会让产品从"诚实的不工作"变成"撒谎的工作"。后者对用户的伤害更大。**

建议把 v0.6.0 拆成两个版本：
- **v0.5.1**（0.5天）：只做失败分桶 + 根因诊断报告。不修任何东西。
- **v0.6.0**（1天）：基于诊断数据定点修复 + 跑 F1 对比 + 用真实业务项目（非靶场）验证。
